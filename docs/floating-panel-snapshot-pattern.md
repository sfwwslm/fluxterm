# 浮动面板快照模式设计

## 背景

FluxTerm 的浮动面板不是主窗口里“搬出去的一块 React 子树”，而是一个新的 Tauri Webview 窗口。  
这意味着浮动窗口会重新初始化自己的 React 状态、Hook、`localStorage` 读取和事件订阅。

如果某个面板直接在浮动窗口里复用主窗口同一套状态 Hook，会出现两个问题：

1. 浮动窗口拿不到主窗口当前的运行时真相，只能从自己的初始状态启动。
2. 如果这个 Hook 还带有持久化或跨窗口广播能力，浮动窗口的空状态可能反向覆盖主窗口已有状态。

AI 面板就曾出现过这个问题：浮动窗口重新实例化 `useAiState` 后，因为没有共享主窗口会话上下文，最终把空聊天记录广播回主窗口。

## 目标

为所有“依赖主窗口运行时上下文”的浮动面板建立统一模式：

1. 主窗口是唯一事实源。
2. 浮动窗口只渲染主窗口广播的快照，不自己维护核心业务状态。
3. 浮动窗口产生的用户操作通过消息代理回主窗口执行。
4. 协议只暴露最小必要状态，避免把主窗口内部实现细节泄漏到浮窗。

## 适用场景

适合使用快照模式的面板：

1. 状态依赖当前活动会话、连接生命周期、工作区焦点或其他主窗口运行时上下文。
2. 行为需要复用主窗口已有命令入口，而不是在浮窗里重复接入底层能力。
3. 状态不适合多窗口并发写入，或者写入冲突成本很高。

当前仓库中的适用例子：

1. `files` 面板
2. `history` 面板
3. `ai` 面板

## 模式分层

### 1. 协议层

每个浮动面板在 `src/features/<domain>/core/floatingSync.ts` 定义自己的协议：

1. `CHANNEL` 常量
2. `Snapshot` 类型
3. `ActionMessage` 类型
4. `SnapshotMessage` 类型
5. `Message` 联合类型

协议要保持“最小但完整”：

1. 快照只包含渲染所需字段。
2. 动作消息只表达用户意图，不重复封装主窗口已经拥有的上下文。
3. 不把内部 reducer、ref、缓存键直接暴露到协议中。

### 2. 主窗口

主窗口负责两件事：

1. 把真实业务状态压缩成快照并广播。
2. 接收浮动窗口动作消息，并调用主窗口已有 action 执行。

主窗口必须满足：

1. 同一时刻只存在一份核心业务状态。
2. 浮动窗口不直接触达底层命令、连接句柄或本地持久化真相。
3. 快照变化时应重新广播，确保浮窗 UI 与主窗口一致。

### 3. 浮动窗口

浮动窗口只做三件事：

1. 打开时请求一次最新快照。
2. 接收快照并转成只读 UI 状态。
3. 把输入、点击、发送、取消等动作通过消息发回主窗口。

浮动窗口不应该：

1. 自己重新初始化核心状态 Hook。
2. 自己做会话级持久化写入。
3. 把自己的局部空状态广播成全局真相。

## AI 面板修复后的实现

AI 面板现在采用与 `files/history` 一致的快照模式：

1. 主窗口中的 `useAiState` 保持唯一真实状态。
2. `src/features/ai/core/floatingSync.ts` 定义 AI 面板快照与动作协议。
3. `AppShell` 在主窗口广播 `messages`、`draft`、`pending`、`errorMessage` 等快照。
4. 浮动 AI 面板只消费快照，并把 `set-draft`、`send`、`cancel`、`clear` 动作回传给主窗口。
5. `useAiState` 增加 `enabled` 开关，避免浮动窗口继续注册 AI 事件监听、读写本地持久化和跨窗口同步。

其中第 5 点很关键。  
如果某个核心 Hook 默认会：

1. 订阅后端事件
2. 读写 `localStorage`
3. 通过 `BroadcastChannel` 做状态同步

那么在浮动窗口里必须显式禁用，避免形成第二个“伪事实源”。

## 推荐落地步骤

新面板接入浮动快照模式时，按下面顺序实现：

1. 先判断该面板是否依赖主窗口运行时真相。
2. 若依赖，先定义 `floatingSync.ts` 协议。
3. 在主窗口新增广播快照和接收动作的 `BroadcastChannel` 逻辑。
4. 在浮动窗口新增请求快照和发送动作的代理逻辑。
5. 把面板 props 改为消费“主窗口态或浮窗快照态”的统一 ViewModel。
6. 检查原 Hook 是否需要增加 `enabled`、`readonly` 或 `sourceOfTruth` 之类的保护开关。

## 不建议这样做

以下做法应避免：

1. 在浮动窗口里直接复用主窗口核心 Hook，并期待它天然共享状态。
2. 通过 `localStorage` 猜测另一窗口的运行时真相。
3. 让两个窗口都能直接写同一份业务状态，再用广播相互覆盖。
4. 为了掩盖后端/状态源问题，在浮窗 UI 层额外加一层临时补丁。

## 总结

浮动面板的核心原则不是“把组件弹出去”，而是“把主窗口状态投影到另一个窗口”。  
只要某个面板依赖主窗口运行时上下文，就应优先采用“主窗口单一事实源 + 浮窗快照 + 动作代理”的模式。
