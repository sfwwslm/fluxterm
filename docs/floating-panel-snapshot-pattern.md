# 浮动面板快照模式设计

## 背景

FluxTerm 的浮动面板并非主窗口中同一棵 React 子树的视觉迁移，而是独立的 Tauri Webview 窗口。浮动窗口会重新初始化自身的 React 状态、Hook、`localStorage` 读取与事件订阅。

在该前提下，若浮动窗口直接复用主窗口的核心状态 Hook，通常会带来以下风险：

1. 浮动窗口无法直接获得主窗口当前运行时状态，只能从本地初始值启动
2. 带有持久化或跨窗口广播能力的 Hook 可能将浮动窗口的初始状态反向写回主窗口

因此，依赖主窗口运行时上下文的浮动面板需要采用统一的快照同步模式。

## 设计目标

1. 主窗口作为唯一事实源
2. 浮动窗口仅渲染主窗口广播的快照，不维护核心业务状态
3. 浮动窗口产生的用户操作通过动作代理回传主窗口执行
4. 协议仅暴露最小必要状态，避免泄漏主窗口内部实现细节

## 适用场景

适合采用快照模式的面板包括：

1. 状态依赖当前活动会话、连接生命周期、工作区焦点或其他主窗口运行时上下文
2. 行为需要复用主窗口已有命令入口，而不适合在浮窗中重复接入底层能力
3. 状态不适合多窗口并发写入，或写入冲突成本较高

当前仓库中的代表性场景：

1. `files` 面板
2. `history` 面板
3. `ai` 面板

## 模式分层

### 1. 协议层

每个浮动面板在 `src/features/<domain>/core/floatingSync.ts` 中定义自己的协议：

1. `CHANNEL` 常量
2. `Snapshot` 类型
3. `ActionMessage` 类型
4. `SnapshotMessage` 类型
5. `Message` 联合类型

协议设计原则：

1. 快照仅包含渲染所需字段
2. 动作消息仅表达用户意图，不重复封装主窗口已掌握的上下文
3. 不将内部 reducer、ref、缓存键等实现细节暴露到协议中

### 2. 主窗口

主窗口负责以下职责：

1. 将真实业务状态压缩为快照并广播
2. 接收浮动窗口动作消息，并调用主窗口既有 action 执行

主窗口侧约束：

1. 同一时刻仅存在一份核心业务状态
2. 浮动窗口不直接触达底层命令、连接句柄或本地持久化真相
3. 快照变化后应重新广播，以保持浮窗与主窗口一致

### 3. 浮动窗口

浮动窗口负责以下职责：

1. 打开时请求最新快照
2. 接收快照并转换为只读 UI 状态
3. 将输入、点击、发送、取消等动作通过消息发送回主窗口

浮动窗口侧约束：

1. 不重新初始化核心状态 Hook
2. 不执行会话级持久化写入
3. 不将局部初始状态广播为全局状态

### 4. 窗口壳与菜单归属

1. 浮动窗口是业务快照消费者，而非窗口编排者；App 级菜单归属固定为 Main
2. 浮动窗口中的配置操作应以动作意图形式回传 Main，由 Main 更新全局状态并广播新快照
3. 浮动窗口不得直接安装或覆盖 App 级菜单
4. 平台外观可存在差异，但不改变“Main 单一事实源 + 浮窗动作代理”的契约

## AI 面板实现说明

AI 面板当前采用与 `files`、`history` 一致的快照模式：

1. 主窗口中的 `useAiState` 维护唯一真实状态
2. `src/features/ai/core/floatingSync.ts` 定义 AI 面板快照与动作协议
3. `AppShell` 在主窗口广播 `messages`、`draft`、`pending`、`errorMessage` 等快照
4. 浮动 AI 面板仅消费快照，并将 `set-draft`、`send`、`cancel`、`clear` 动作回传主窗口
5. `useAiState` 提供 `enabled` 开关，用于禁用浮动窗口中的事件监听、持久化写入与跨窗口同步

当某个核心 Hook 默认具备以下能力时，应在浮动窗口中显式禁用：

1. 后端事件订阅
2. `localStorage` 读写
3. 基于 `BroadcastChannel` 的状态同步

此举用于防止形成第二个业务事实源。

## 接入步骤

新面板接入浮动快照模式时，建议按以下顺序实现：

1. 判断该面板是否依赖主窗口运行时上下文
2. 若依赖，则先定义 `floatingSync.ts` 协议
3. 在主窗口实现快照广播与动作接收逻辑
4. 在浮动窗口实现快照请求与动作代理逻辑
5. 将面板 props 统一为消费“主窗口态或浮窗快照态”的 ViewModel
6. 评估原有 Hook 是否需要增加 `enabled`、`readonly` 或 `sourceOfTruth` 等保护开关

## 反模式

以下做法不属于推荐实现：

1. 在浮动窗口直接复用主窗口核心 Hook，并假设两者天然共享状态
2. 通过 `localStorage` 推断另一窗口的运行时状态
3. 让两个窗口同时写入同一份业务状态，再依靠广播相互覆盖
4. 通过浮窗 UI 层临时补丁替代对状态源问题的修正

## 总结

浮动面板设计的核心不是“将组件弹出主窗口”，而是“将主窗口状态投影到独立窗口”。凡是依赖主窗口运行时上下文的面板，均应优先采用“主窗口单一事实源 + 浮窗快照 + 动作代理”的模式。
