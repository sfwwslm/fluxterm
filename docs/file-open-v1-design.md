# 文件打开与远端编辑设计

## 目标

为文件管理器提供统一的文件打开能力，并在远端文件场景下提供可回传的远端编辑工作流。

覆盖两类场景：

1. 本地文件管理器中双击普通文件后，直接在本机打开
2. SSH 文件管理器中双击普通文件后，创建受管工作副本，使用本地编辑器打开，并在可跟踪场景下提供回传链路

## 当前能力范围

### 已支持

1. 本地普通文件双击打开
2. 远端普通文件下载到本地工作副本后打开
3. 提供全局默认编辑器配置
4. 默认编辑器不可用时回退到系统默认程序
5. 文本类远端文件支持本地修改检测与确认回传
6. 回传前执行远端文件变更校验，避免覆盖更新后的远端内容
7. 远端编辑实例支持集中索引与会话内状态管理
8. 打开失败或回传失败时显示横幅提示并记录日志

### 当前边界

1. 不定义拖拽打开或批量打开行为
2. 不对所有文件类型提供变更跟踪能力
3. 二进制文件或超出限制的文件仅执行“打开”，不启用回传跟踪
4. 缓存清理仅提供基础保留策略，不承担精细化生命周期管理

## 交互语义

### 本地文件管理器

1. 双击目录：进入目录
2. 双击普通文件：直接打开文件
3. 单击：仅用于选中

### SSH 文件管理器

1. 双击目录：进入目录
2. 双击普通文件：
   - 后端准备远端编辑工作副本
   - 调用本地编辑器或系统默认程序打开
   - 若文件属于可跟踪文本文件，则登记远端编辑实例
3. 打开失败时显示横幅提示

## 默认编辑器配置

### 配置位置

默认编辑器属于本机行为，不属于会话行为，因此配置位于“设置”而非“会话设置”。

### 配置键

- `file.defaultEditorPath`

### 打开优先级

1. 若配置了默认编辑器路径，且程序存在并可启动，则优先使用该编辑器
2. 默认编辑器未配置或启动失败时，回退到系统默认程序
3. 两者均失败时，显示失败横幅并记录错误日志

### 配置交互

当前交互采用“选择程序”模式：

1. 通过“选择程序”选择可执行文件
2. 通过“清空”恢复为系统默认程序打开
3. 配置值持久化到 `file.defaultEditorPath`

## 本地文件打开实现

### 前端职责

1. 判断双击目标是否为普通文件
2. 调用统一的 `file_open` 命令封装
3. 打开失败时显示横幅

### 后端职责

1. 根据配置决定是否优先使用默认编辑器
2. 校验默认编辑器路径是否存在
3. 执行程序打开动作
4. 在默认编辑器不可用时回退到系统默认程序

## 远端编辑实现

### 打开流程

1. 用户在 SSH 文件管理器中双击普通文件
2. 后端通过 `remote_edit_open` 执行远端文件一致性检查
3. 为目标文件准备本地工作副本目录
4. 复用默认编辑器或系统默认程序打开本地工作副本
5. 若文件满足文本跟踪条件，则登记远端编辑实例并启动后台监听

### 工作副本目录

缓存根目录位于应用数据目录下：

- `appData/cache/remote-files/`

当前工作副本采用稳定目录结构：

- `appData/cache/remote-files/active/{host}__p{port}/{remoteDir...}/{fileName}__{pathHash}/{fileName}`

该结构同时满足：

1. 同一远端文件复用同一工作副本目录
2. 不同主机、端口和远端路径之间保持隔离
3. 目录结构具备可读性，便于人工排查

### 集中索引

远端编辑实例通过集中索引维护：

- `appData/cache/remote-files/index.json`

索引记录以下信息：

1. `instanceId`
2. `sessionId`
3. 远端目标标识（主机、用户、端口）
4. `remotePath`
5. `fileName`
6. `downloadedAt`
7. `remoteMtime`
8. `remoteSize`
9. `trackChanges`
10. 工作副本基线快照

该索引用于：

1. 复用既有工作副本
2. 恢复远端编辑实例真相
3. 判断本地副本是否存在未回传修改

### 工作副本复用规则

打开同一远端文件时，后端按以下顺序判断：

1. 始终先获取远端 `stat`
2. 若本地工作副本存在未回传修改，则拒绝直接复用
3. 若本地基线与远端版本一致，则复用既有工作副本
4. 其余情况重新下载远端文件并覆盖本地工作副本

### 可跟踪文件范围

远端编辑跟踪仅对可识别的文本类文件启用，并受文件大小上限控制。

当前规则包括：

1. 目标必须是普通文件
2. 文件大小不得超过 `2 MB`
3. 文件扩展名需在白名单内，例如 `txt`、`md`、`json`、`ts`、`tsx`、`rs`、`py`、`sh` 等

不满足条件的远端文件仍可打开，但不登记跟踪实例，也不提供回传提示。

## 变更检测与回传

### 本地修改检测

对于已启用跟踪的远端编辑实例，后端会：

1. 启动后台轮询任务
2. 定期读取本地工作副本快照
3. 使用静默窗口避免编辑器写入过程中的抖动误判
4. 当检测到稳定修改后，将实例状态置为 `pending_confirm`

当前关键常量：

1. 轮询间隔：`REMOTE_EDIT_POLL_INTERVAL_MS = 2000`
2. 静默窗口：`REMOTE_EDIT_QUIET_WINDOW_MS = 1200`

### 回传确认

本地修改进入待确认状态后：

1. 前端收到 `remote-edit:update`
2. 用户可选择确认上传，或忽略本次待确认修改
3. 主窗口支持会话级自动确认，减少同一会话内重复确认

### 回传校验

确认上传时，后端通过 `remote_edit_confirm_upload` 执行：

1. 再次读取远端文件 `mtime` 与 `size`
2. 若远端文件已发生变化，则返回冲突错误并终止回传
3. 若远端状态一致，则上传本地工作副本
4. 上传成功后刷新远端基线与本地基线

### 忽略待确认修改

用户忽略待确认修改时：

1. 当前待确认快照被清除
2. 当前内容哈希加入忽略判定
3. 相同内容不会立即重复触发确认

## 状态模型

远端编辑实例当前使用以下状态：

- `synced`
- `pending_confirm`
- `uploading`
- `sync_failed`

状态快照对前端暴露：

1. `instanceId`
2. `sessionId`
3. `remotePath`
4. `localPath`
5. `fileName`
6. `downloadedAt`
7. `remoteMtime`
8. `remoteSize`
9. `trackChanges`
10. `status`
11. `lastSyncedAt`
12. `lastErrorCode`
13. `lastError`

## 浮动文件面板

浮动文件面板中的“双击打开文件”统一由主窗口代理处理，不在浮动窗口内独立执行。

该设计用于确保：

1. 主窗口与浮动窗口行为一致
2. 文件打开与远端编辑链路保持单一事实源
3. 浮动窗口不维护独立的远端编辑状态机

## 缓存清理策略

缓存目录提供基础清理规则，用于控制长期增长。

### 清理原则

1. 应用启动或首次使用时保留缓存目录，不执行全量清空
2. 远端编辑索引与工作副本由后端统一维护
3. 清理不影响仍存在的活动工作副本
4. 过期工作副本按保留期清理

### 当前保留期

- `7` 天

### 清理结果

清理后会同步裁剪失效索引项，并记录遥测日志。

## 日志与事件

### 前端 / 窗口事件

- `remote-edit:update`

### 遥测事件

当前链路中包含以下典型事件：

1. `remote_edit.open.prepared`
2. `remote_edit.open.completed`
3. `remote_edit.local_change_detected`
4. `remote_edit.upload.started`
5. `remote_edit.upload.succeeded`
6. `remote_edit.upload.failed`
7. `remote_edit.upload.dismissed`
8. `remote_edit.cache.cleanup`

## 与文件管理器交互的关系

文件打开与远端编辑能力仅扩展普通文件双击行为，不改变目录与选择语义：

1. 目录：双击进入
2. 文件：双击打开
3. 单击：选中
4. 右键：先选中再弹出菜单
