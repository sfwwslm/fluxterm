# 终端与文件管理器路径联动设计

## 目标

在 SSH 会话中，让终端当前工作目录与文件管理器目录保持联动。

当前目标范围：

- 支持 bash 常见提示符
- 当前仅支持 bash
- 不修改服务端配置
- 不向服务端安装 agent
- 不破坏终端原生交互体验

## 最终方案

当前实现采用“解析终端 prompt”方案，而不是向远端额外注入命令。

前端在终端输出流中解析 bash 常见提示符：

- `user@host:/abs/path$`
- `user@host:~/path$`
- `/abs/path$`
- `~/path$`

解析结果会上报两类信息：

- prompt 中显示的当前用户名
- prompt 中显示的当前路径

应用层再把这两类信息与当前 SSH 会话的初始登录身份组合起来，决定是否继续同步到 SFTP 文件管理器。

## 用户配置

当前提供会话级开关：

- `终端路径联动文件管理器`

配置行为：

- 开启后，尝试根据支持的 bash prompt 解析路径并同步到文件管理器
- 关闭后，不再继续解析 prompt，也不再驱动文件管理器路径
- 该配置写入 `session.json`，并对所有终端会话统一生效

需要明确的是：

- “开启联动”不等于“当前会话一定支持联动”
- 当前实现仍然只支持 bash，zsh、本地 shell、以及其他无法稳定解析的 prompt 会显示为不支持联动

## 为什么放弃命令注入方案

实现过程中尝试过两类主动探测：

1. 用户执行 `cd` 后自动追加 `pwd`
2. 在 shell 中安装 `PROMPT_COMMAND` / `precmd` hook

这两类方案都存在同一个核心问题：它们都会把内部实现细节带进真实终端交互。

已验证的副作用包括：

- 终端里出现内部探测命令回显
- 首屏 prompt 与欢迎信息边界被打乱
- 输出流抑制逻辑容易误伤正常 prompt
- 不同 shell/服务端环境下切分点不稳定

结论是：在交互 shell 已经启动后，再发送内部探测命令，稳定性不够，且容易破坏用户感知。

因此当前设计明确改为：

- 不再向远端注入任何路径探测命令
- 只消费终端已经自然产生的 prompt 文本

## 为什么只支持 bash

bash 的常见提示符通常能给出：

- 绝对路径
- 或以 `~` 为前缀的家目录路径

这两种信息都可以稳定展开为 SFTP 可访问路径。

zsh 默认提示符常见形态例如：

- `root@MacBook-Pro ~ %`
- `root@MacBook-Pro codes %`
- `root@MacBook-Pro flux-term %`

这类提示符的问题是：

- 结尾通常是 `%`
- 家目录内常显示缩略路径段
- 很多情况下不包含绝对路径，也不包含足够的父路径信息

因此即使识别出“这是一个 prompt”，也无法稳定还原出 SFTP 应该进入的绝对目录。

基于这个限制，当前设计选择：

- bash 支持联动
- zsh、sh、以及其他无法稳定解析的 prompt 统一视为不支持
- 当前会话只记录一条日志，之后停止继续解析

相关日志事件：

- `terminal:cwd-sync-unsupported-prompt`

## 文件管理器中的联动状态

文件管理器路径栏前会展示当前会话的联动状态图标。

当前共有 4 种状态：

- `联动中`

1. 当前会话已成功解析出受支持的 bash prompt
2. 且当前 shell 身份与 SFTP 登录身份一致

- `联动已暂停`

1. 当前 prompt 用户名与 SSH 初始登录用户名不一致
2. 常见于终端内执行了 `su` 之类的身份切换

- `不支持联动`

1. 功能开关是开启的
2. 但当前会话不满足联动条件，例如本地 shell、zsh、sh 或其他无法稳定解析的 prompt

- `联动已关闭`

1. 用户在会话设置中主动关闭了路径联动功能

这 4 种状态的区别是：

- `不支持联动`：能力开着，但当前会话不支持
- `联动已关闭`：用户主动关闭功能

## 用户身份一致性保护

即使 bash prompt 可以解析出路径，也不代表 SFTP 一定应该跟随。

典型反例是：

- SSH 初始登录用户是普通用户
- 终端里执行 `su root`
- prompt 用户变为 `root`

这时终端 shell 与 SFTP 子系统已经不再共享同一个用户身份。

如果继续联动，会出现：

- `cd ~` 被错误展开到普通用户家目录
- 或终端能进入的目录，SFTP 没权限进入

为避免这个问题，当前实现加入了“prompt 用户名与 SSH 初始用户名一致性”校验：

- 一致：允许联动
- 不一致：暂停联动
- 再次恢复一致：自动恢复联动

相关日志事件：

- `terminal:cwd-sync-paused-user-mismatch`
- `terminal:cwd-sync-resumed-user-match`

## `~` 的处理

prompt 中的 `~` 不能直接拿来访问 SFTP。

当前实现做法是：

1. 当会话第一次解析到 `~`，并且文件管理器当前已经处于一个绝对路径时，将该绝对路径记为当前会话的 home
2. 后续遇到 `~` 或 `~/subdir` 时，再把它展开成绝对路径

这意味着：

- 绝对路径始终优先
- `~` 展开依赖当前会话已知的 home
- 如果 home 尚未知，`~` 不会强行同步

## 首个 SSH 会话的冷启动时序

实现过程中还遇到过一个只在“主窗口里文件管理器可见”时出现的首会话竞态：

- 应用冷启动后建立第一个 SSH 会话
- 终端横幅没有正常显示
- 文件管理器停留在 `正在检测 SFTP`
- 但后续再建立新的 SSH 会话时一切正常

根因不在 prompt 解析本身，而在后端会话状态语义：

- 旧实现会在 SSH 认证成功后立即发出 `connected`
- 此时交互 shell 的 `PTY` 与 `request_shell` 还未完全就绪
- 前端一旦看到 `connected`，就会在文件管理器可见时立即触发首轮 `sftp_home`
- 首个会话冷启动阶段，这轮 SFTP 初始化可能抢在终端首屏横幅与 prompt 输出之前进入后端会话循环

这会导致两个直接问题：

- 终端首屏横幅/提示符显示异常
- 文件管理器的 `checking` 状态无法按预期完成

当前修正方式是直接调整后端状态时序：

- 只有在 `channel_open_session`
- `request_pty`
- `request_shell`

全部完成之后，才向前端发出 `connected`

这样前端继续沿用现有的 `connected -> 首轮 SFTP 初始化` 逻辑即可，不需要再在前端增加额外的“等待首段输出”之类补丁逻辑。

这个取舍的设计结论是：

- `connected` 必须表示“交互 shell 已经真正可用”
- 不应把“SSH transport 已连上”误当成“终端和 SFTP 都可以开始工作”
- 这类时序语义应在后端状态机中定义，而不是在前端靠补丁猜测

## 已知边界

- 本地 shell 当前统一视为不支持联动
- 不支持 zsh 默认缩略路径提示符联动
- 不支持高度自定义的 PS1 / prompt 主题
- 不支持终端用户与 SFTP 用户不一致时继续联动
- 只保证“终端驱动文件管理器”，不做反向驱动 shell cwd
